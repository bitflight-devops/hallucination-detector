## 2. Does `evidenceSnippets` embed the matched evidence phrase into the reason string?

Yes — unconditionally and verbatim.

```javascript

  .slice(0, 6)
  .map((m) => `- ${m.kind}: "${m.evidence}"`)
  .join('\n');
```

When `findTriggerMatches()` fires on the word `since`, it pushes:

```javascript

```

`evidenceSnippets` then becomes:


This string is interpolated directly into `reason` at line 711:

```json

  "reason": "Hallucination-detector STOP HOOK blocked this response.\n\nDetected trigger language in your last assistant message:\n- causality_language: \"since\"\n\n..."

```

**The word "since" appears inside the `reason` field of the block output.**

## 3. Could `getLastAssistantText()` pick up the hook's own prior block message?

This depends on what Claude Code writes to the transcript after a block.

`getLastAssistantText()` (lines 86–102) accepts any transcript entry where:

- `entry.type === 'assistant'`
- `entry.message` exists and has non-empty `content`
- `entry.isSidechain` is falsy

`hooks/hooks.json` registers the stop hook as a `"type": "command"` hook. The Claude Code documentation describes stop hooks as receiving a `stop_hook_active` flag when the assistant is being re-prompted after a block — meaning Claude Code does re-invoke the assistant after a block.

The question is whether Claude Code writes the hook's `reason` text back into the transcript as an assistant message. If it does — if the injected framing/correction prompt is stored with `type: "assistant"` — then `getLastAssistantText()` would read it on the next invocation.

**From the code alone, this cannot be confirmed or refuted.** `getLastAssistantText()` only filters on `type === 'assistant'` and `isSidechain`. If Claude Code stores the block reason as a `type: "human"` or `type: "system"` entry, it would not be picked up. If it stores it as `type: "assistant"` (e.g., as part of a synthetic assistant turn that contains the rejection text), it would be picked up and scanned.

## 4. Confirmed or refuted: does the hook's own output contain "since"?

**Confirmed — conditionally.**

When `findTriggerMatches()` matches `since` in the assistant's text, the evidence phrase `"since"` is embedded verbatim into `evidenceSnippets`, which is interpolated into `reason`. The block JSON emitted to stdout contains the literal string `since` inside the `reason` field.

Whether this causes the self-triggering loop depends on whether Claude Code writes the block `reason` into the transcript as an `assistant`-typed entry. If it does, the next hook invocation will call `getLastAssistantText()`, read the reason string containing `since`, and fire `causality_language: "since"` again — producing the loop described in the hypothesis.

## 5. Summary of the attack surface

The self-trigger loop has two required conditions, both of which are present:

| Condition                                                                | Status                                                                                  |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------- |
| Hook output embeds the flagged evidence phrase verbatim                  | **Confirmed** (line 704: `"${m.evidence}"`)                                             |
| `getLastAssistantText()` could read the block reason from the transcript | **Cannot confirm or refute from code alone — depends on Claude Code transcript format** |

The fix surface is clear regardless: `evidenceSnippets` should not embed the raw evidence phrase as a scannable string in the emitted reason. Wrapping it in backticks (as the reason text itself already instructs the assistant to do, line 718) would suppress re-detection via `stripLowSignalRegions()`, which removes inline code spans before scanning.
